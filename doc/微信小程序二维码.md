# 舞萌排卡系统开发文档补充 v0.2

## 1. 本次新增需求

在原有 Web 排卡系统基础上，后续需要支持：

1. 管理员为机厅、机台或队列生成快捷 NFC 标签。
2. 管理员为机厅、机台或队列生成微信小程序二维码。
3. NFC 标签和小程序二维码本质上都指向一个包含机厅 ID / 队列 ID 的 API 链接。
4. 用户扫码或触碰 NFC 后，根据当前排队状态进入不同操作流程。
5. 支持“即将上机确认”，超过自定义时间未确认则自动过号。
6. 用户下机后，系统询问是否需要重新排队。

---

## 2. NFC / 小程序二维码设计

### 2.1 使用场景

管理员可以在后台为以下对象生成快捷入口：

- 机厅
- 机台
- 队列

实际落地方式：

```txt
1. NFC 标签：写入一个 URL。
2. 微信小程序二维码：携带 scene 参数。
3. 普通二维码：指向 Web 链接。
````

例如：

```txt
https://example.com/q/arcade_123
https://example.com/q/queue_456
https://example.com/q/machine_789
```

也可以使用统一跳转 API：

```txt
https://example.com/api/v1/quick-entry?type=queue&id=queue_456
```

推荐使用短链接形式：

```txt
https://example.com/s/AbC123
```

短链接再映射到具体资源：

```json
{
  "type": "queue",
  "targetId": "queue_456",
  "arcadeId": "arcade_123",
  "machineId": "machine_1"
}
```

## 3. 快捷入口类型

### 3.1 机厅入口

适合贴在机厅门口、前台、墙面。

```txt
https://example.com/a/arcade_123
```

用户进入后看到：

```txt
1. 机厅信息
2. 当前机台列表
3. 当前队列列表
4. 每个队列的等待人数
5. 可选择具体队列排队
```

### 3.2 机台入口

适合贴在某一台舞萌机旁边。

```txt
https://example.com/m/machine_123
```

用户进入后看到：

```txt
1. 当前机台信息
2. 当前绑定队列
3. 当前队伍状态
4. 快速排队 / 查看队伍
```

### 3.3 队列入口

适合直接进入某个队列。

```txt
https://example.com/q/queue_123
```

用户进入后，系统直接判断用户状态并显示对应操作。

## 4. 用户状态判断流程

用户通过 NFC / 二维码进入后，后端需要判断：

```txt
1. 用户是否登录。
2. 当前入口对应哪个机厅 / 机台 / 队列。
3. 用户是否已经在该队列中。
4. 用户当前排卡状态是什么。
5. 根据状态返回对应操作。
```


规则：

```txt

5. 过号后状态变为 skipped。
```

自定义时间示例：


## 6. 状态机设计

### 6.1 QueueEntry 状态

```txt
not_in_queue
waiting
called
playing
finished
skipped
cancelled
```

### 6.2 状态流转

```txt
not_in_queue -> waiting

waiting -> cancelled
waiting -> called

called -> playing
called -> skipped
called -> cancelled

playing -> finished

finished -> waiting
skipped -> waiting
cancelled -> waiting
```

说明：

```txt
finished -> waiting 表示重新排队。
skipped -> waiting 表示过号后重新排队。
cancelled -> waiting 表示取消后重新排队。
```

### 8.5 NFC 标签写入内容

管理员后台展示：

```txt
NFC 写入链接：
https://example.com/s/AbC123
```

管理员可以使用手机 NFC 工具写入该链接。

系统本身不一定需要直接写 NFC，因为浏览器通常不能稳定操作所有 NFC 标签。

如果后续需要 Web NFC，可增加：

```txt
仅支持 Android Chrome 的 Web NFC 写入能力。
iOS Safari 暂不适合作为主要写入方案。
```

推荐 MVP 做法：

```txt
1. 后台生成链接。
2. 后台生成二维码。
3. 管理员复制链接。
4. 管理员使用 NFC 写入工具写入标签。
```

## 11. 队列规则配置补充

Queue 表中的 `ruleConfig` 建议包含：

```json
{
  "callTimeoutSeconds": 180,
  "allowUserCancel": true,
  "allowUserFinish": true,
  "allowUserConfirmPlaying": true,
  "allowRejoinAfterFinished": true,
  "allowRejoinAfterSkipped": true,
  "allowRejoinAfterCancelled": true,
  "autoSkipWhenTimeout": true,
  "maxActiveEntryPerUser": 1,
  "maxGroupSize": 2
}
```

字段说明：

| 字段                          | 说明           |
| --------------------------- | ------------ |
| `callTimeoutSeconds`        | 叫号后多少秒内必须确认  |
| `allowUserCancel`           | 是否允许用户自行取消排队 |
| `allowUserFinish`           | 是否允许用户自行点击下机 |
| `allowUserConfirmPlaying`   | 是否允许用户自行确认上机 |
| `allowRejoinAfterFinished`  | 完成后是否允许重新排队  |
| `allowRejoinAfterSkipped`   | 过号后是否允许重新排队  |
| `allowRejoinAfterCancelled` | 取消后是否允许重新排队  |
| `autoSkipWhenTimeout`       | 超时是否自动过号     |
| `maxActiveEntryPerUser`     | 单用户最大活跃排卡数   |
| `maxGroupSize`              | 单次排卡最大人数     |

## 13. 自动过号任务设计

### 13.1 方案一：定时扫描

每 10 秒扫描一次：

```sql
SELECT *
FROM queue_entries
WHERE status = 'called'
AND confirm_deadline < NOW();
```

然后执行：

```txt
called -> skipped
```

优点：

```txt
简单可靠，适合 MVP。
```

缺点：

```txt
不是毫秒级精确。
```

---

### 13.2 方案二：Redis 延迟任务

叫号时写入 Redis 延迟任务：

```txt
queue:auto-skip:entry_123
```

到期后执行自动过号。

优点：

```txt
更实时。
```

缺点：

```txt
实现复杂度更高。
```

MVP 推荐：

```txt
先使用定时扫描。
```

## 20. 总结

本次补充后，系统的核心体验变为：

```txt
NFC / 二维码作为线下快捷入口
        ↓
根据用户当前状态自动展示最合适的操作
        ↓
未排队：排队 / 查看信息
已排队：取消 / 查看信息
即将上机：确认 / 超时过号
正在上机：下机 / 重新排队
```

这套设计适合线下机厅实际使用：

```txt
1. 管理员只需要贴一个 NFC 标签或二维码。
2. 玩家扫码即可进入当前队列。
3. 系统自动判断玩家状态。
4. 玩家不用在复杂页面里找按钮。
5. 后续可以平滑接入微信小程序。
```
